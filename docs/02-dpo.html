Chapter 2

DPO: learning from a preference

Direct Preference Optimization is the simplest way to align a model. You never write a reward function and you never run the model during training. You just hand it pairs of answers — a chosen one and a rejected one — and it adjusts itself to prefer the chosen. The interesting part is how it does that, which the chart below catches red-handed.

What one training example looks like

Each pair uses the same question. The chosen answer is the curated reference — verdict first, then a tight rationale, then the disclaimer. The rejected answer is what the raw base model produced for that same question before any finetuning: it continues the abstract in the paper's own voice and never actually commits to a verdict.

Chosen — curated reference
yes In this population, neither BMI nor neck circumference nor waist circumference is predictive of autoCPAP pressure. Therefore, the previously derived algorithm does not adequately predict the fixed pressure… [+ standard disclaimer]
Rejected — raw base output
[IMPACT] The study is a continuation of the previous research, which showed the correlation between obesity and OSA severity. The authors wanted to find out whether this relationship remains in the mo…

This is a genuinely easy preference — the two answers barely resemble each other. That matters, and Chapter 4 comes back to it. For now, watch what the model does with it.

How DPO actually improves: it pushes the bad answer down

DPO tracks a reward for each side of the pair — really just how much more (or less) likely the model now finds that answer, compared to the frozen SFT reference. Intuition says training should make the chosen answer more attractive. But the SFT model already liked the curated answers, so there was little room to push them up. Instead, almost all the work goes into shoving the rejected answer's reward through the floor:

The chosen line barely moves; the rejected line falls off a cliff. This is the most important intuition about DPO: it separates chosen from rejected by whatever route is cheapest, and when the reference already prefers the chosen answer, the cheapest route is to punish the rejected one. By step 15 the model already prefers chosen over rejected on 100% of pairs — the rest of training just widens an already-decided gap.

An easy preference teaches an easy lesson. Because "rejected" here is raw base text — formatless, verdict-free — the model mostly learns "don't sound like the un-finetuned model." That's a style-and-format lesson, not a "get the medical verdict right" lesson. Keep that in mind when Chapter 4 shows DPO's verdict accuracy came out flat: it was never really asked to improve verdicts.

The config, in brief

QLoRA (4-bit base + a rank-16 adapter), pref_beta: 0.1, learning rate 5e-6 — about a tenth of the SFT rate, because preference nudges should be gentle — for one epoch over the 800 pairs. Around nine minutes on the 4090. The full config is configs/dpo.yaml.
