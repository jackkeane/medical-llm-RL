Chapter 1

The shared starting line

To compare two alignment methods fairly, they must start from exactly the same model. This chapter builds that shared starting point — a merged SFT model — and lays out the one real difference the rest of the walkthrough is about: what kind of signal each method learns from.

Step one: freeze the SFT model into a single starting point

The finetuning project left us a LoRA adapter — a small file of trained weights that rides on top of the frozen base model. For alignment we first merge that adapter into the base, producing one standalone model, sft-merged. It plays two roles at once:

  1. The identical start. Both DPO and GRPO train a brand-new rank-16 adapter on top of sft-merged. Same weights, same rank, same GPU — the only variable left is the learning signal.
  2. DPO's reference model. DPO works by comparing the model it is training against a frozen reference. Because the fresh adapter starts at zero, "the model with its adapter switched off" is exactly sft-merged — so the reference is genuinely the SFT model, which is what makes the preference math correct.

The one real difference: the shape of the signal

SFT learns by imitation — "here is the correct answer, raise its probability." Alignment methods learn from something weaker but cheaper to get: a judgement about answers. DPO and GRPO take two different forms of judgement.

Keep this distinction in hand. DPO is offline — it never runs the model during training, it just adjusts probabilities between two answers it was handed. GRPO is online — it generates answers and reacts to how good they are. That single difference explains almost everything in Chapter 4.

Why align between SFT and compression?

Order matters. Alignment nudges the model's behavior; compression shrinks its weights. If you compressed first and aligned second, you'd be aligning a model you're about to change again. Aligning first means the pruning and quantization in the next project preserve behavior that's already settled.
